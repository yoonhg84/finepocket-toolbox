# toolbox.finepocket.app — AdSense 승인/수익화 개선 계획

> 작성일: 2026-03-30
> 범위: 1) 타이틀/메타 정리, 2) `ads.txt` 및 정책 페이지 보강, 3) 카테고리 허브 페이지 추가
> 목적: AdSense 심사 통과 가능성을 높이고, 고의도 페이지 중심의 광고 수익 구조를 만든다.

---

## 0. 현재 진단 요약

현재 사이트는 아래 기본 조건은 충족한다.

- `robots.txt`, `sitemap.xml` 정상 노출
- `about`, `privacy`, `terms`, `contact` 페이지 존재
- 개별 도구 페이지에 `canonical`, `WebApplication`, `FAQ` 구조화 데이터 존재
- 툴 콘텐츠는 SSR 텍스트와 인터랙티브 UI가 함께 제공됨

하지만 아래 3가지가 심사 및 수익화의 핵심 리스크다.

1. **메타 품질 저하**
   - 개별 툴 페이지의 `<title>`에 사이트명이 중복 렌더링됨
   - 카테고리/홈/정적 페이지 메타 전략이 일관되지 않음

2. **신뢰 신호 부족**
   - `About`, `Privacy`, `Contact`가 너무 짧아 운영 주체와 광고/데이터 처리 정책이 약하게 보임
   - `ads.txt` 부재
   - 향후 광고 도입 시 CMP/쿠키 설명이 부족함

3. **정보 구조 약함**
   - 카테고리 허브가 없어서 툴 목록과 검색 의도를 묶는 정보 페이지가 부족함
   - 헤더 네비게이션은 드롭다운 중심이고, 카테고리 자체를 랜딩 페이지로 설명하지 못함
   - 일부 툴 분류가 사용자 의도와 다소 어긋남

## 1. 목표와 완료 기준

### 목표

- AdSense 심사 시 `thin content`, `low trust`, `unclear structure`로 보일 가능성을 낮춘다.
- 광고주 신뢰와 페이지 문맥을 강화해 금융/고의도 페이지의 RPM 기반을 만든다.
- 툴 페이지 외에 카테고리 레벨의 설명형 페이지를 추가해 내부 링크 구조를 개선한다.

### 최종 완료 기준

- 모든 페이지의 메타 타이틀이 중복 없이 일관된 규칙으로 출력된다.
- `Privacy/About/Contact/Terms`가 광고, 데이터 처리, 운영 주체 관점에서 충분한 신뢰 정보를 제공한다.
- 운영용 `ads.txt` 적용 경로와 배포 절차가 정리된다.
- `/developer`, `/text`, `/finance` 허브 페이지가 생기고, 헤더/브레드크럼/내부 링크가 연결된다.
- 툴 페이지에서 카테고리 링크를 통해 상위 정보 페이지로 이동 가능해진다.

## 2. 작업 스트림 1 — 타이틀/메타 정리

### 문제 정의

현재 툴 페이지 메타 타이틀은 `Free Online Tool | FinePocket Toolbox`를 포함한 값을 다시 루트 `title.template`에 통과시켜 사이트명이 이중으로 붙는다. 이건 심사 탈락 사유는 아니지만, 저품질/저완성 신호로 보일 수 있다.

### 목표

- 타이틀 규칙을 단일화한다.
- 홈, 카테고리, 툴, 정책 페이지 메타를 역할별로 구분한다.
- 향후 카테고리 허브 페이지 추가를 고려한 메타 규약을 만든다.

### 작업 범위

1. 루트 `title.template`와 개별 페이지 `title` 책임 범위를 분리한다.
2. 툴 메타 생성기에서 사이트명 중복을 제거한다.
3. 홈/정적 페이지/카테고리 페이지의 설명문을 현재 구조에 맞게 정리한다.
4. 가능하면 홈과 카테고리 페이지에도 `canonical`을 명시한다.
5. 향후 소셜 미리보기 품질을 위해 `openGraph`/`twitter` 전략을 문서화한다.

### 수정 대상 파일

- `src/app/layout.tsx`
- `src/lib/seo.ts`
- `src/app/page.tsx`
- `src/app/about/page.tsx`
- `src/app/contact/page.tsx`
- `src/app/privacy/page.tsx`
- `src/app/terms/page.tsx`
- 카테고리 허브 페이지 추가 시 해당 `page.tsx`

### 구현 순서

1. 메타 정책 결정
   - 루트는 `%s | FinePocket Toolbox` 템플릿만 담당
   - 개별 페이지는 사이트명을 제외한 순수 제목만 반환
2. 툴 메타 생성기 수정
   - 예: `Loan Calculator - Free Online Tool`
3. 정적 페이지 제목 통일
   - 예: `About`, `Privacy Policy`, `Contact Us`
4. 홈페이지 설명 보강
   - 단순 툴 나열보다 사이트 가치와 카테고리 구조를 반영
5. 카테고리 허브용 메타 초안 작성
   - 예: `Developer Tools`, `Text Tools`, `Finance & Calculators`

### 완료 조건

- 실제 렌더링된 `<title>`에 사이트명이 한 번만 노출된다.
- 툴/정책/카테고리 페이지 제목 규칙이 일관된다.
- 홈/카테고리/정적 페이지의 설명이 역할에 맞게 구분된다.

### 예상 난이도

- 낮음
- 구현 0.5일, 검수 0.5일

## 3. 작업 스트림 2 — `ads.txt` 및 정책 페이지 보강

### 문제 정의

정책 페이지는 존재하지만 광고 운영 관점에서는 아직 초안 수준이다. `ads.txt`가 없고, 개인정보/광고/쿠키/CMP에 대한 운영 설명이 부족하다. AdSense는 단순히 페이지 존재 여부보다 사이트 신뢰성과 정책 정합성을 본다.

### 목표

- 광고 도입 전 필요한 정책·운영 기반을 문서와 페이지에 반영한다.
- 실제 AdSense 연동 시 필요한 `ads.txt` 반영 절차를 준비한다.
- 사용자와 심사자 모두에게 운영 주체와 데이터 처리 방식을 명확히 보여준다.

### 작업 범위

1. `Privacy Policy` 확장
   - 광고(AdSense), 쿠키/로컬 스토리지, 호스팅(Vercel), 로그, analytics 사용 여부, 사용자 권리, 문의 채널 명시
2. `About` 확장
   - 운영 목적, 운영자/브랜드 정보, 업데이트 원칙, 품질 원칙, 광고 원칙 명시
3. `Contact` 보강
   - 문의 목적 분리, 응답 SLA, 광고/비즈니스 문의 경로 명확화
4. `Terms` 보강
   - 계산 결과의 참고용 성격, 면책, 정책 변경 방식 정리
5. `ads.txt` 도입
   - 퍼블리셔 ID 확보 전까지는 문서화
   - ID 발급 후 `public/ads.txt` 또는 동등한 정적 경로 배포
6. CMP 대응 계획 문서화
   - EEA/영국/스위스 대상 광고 시 Google 인증 CMP 필요
   - 정책 페이지에서 이 사실을 사용자 친화적으로 설명

### 수정 대상 파일

- `src/app/about/page.tsx`
- `src/app/privacy/page.tsx`
- `src/app/contact/page.tsx`
- `src/app/terms/page.tsx`
- `public/ads.txt` 또는 동등한 정적 경로
- 필요 시 공통 JSON-LD 유틸/컴포넌트 추가

### 구현 순서

1. `Privacy Policy` 재작성
   - 수집 데이터 없음/있음 구분
   - 광고 파트너, 호스팅, 로그, analytics, 문의 이메일 명시
2. `About` 재작성
   - "왜 이 사이트를 운영하는지", "무엇을 약속하는지" 추가
3. `Contact` 재정리
   - 일반 문의/버그/광고 제안/비즈니스 문의를 구분
4. `Terms` 문구 강화
   - 금융/계산기 결과의 참고용 성격 명확화
5. `ads.txt` 적용 준비
   - 퍼블리셔 ID 입력 위치와 배포 절차 문서화
6. 선택 작업
   - `Organization` JSON-LD 추가
   - 정책 페이지에 마지막 업데이트일 정교화

### `ads.txt` 운영 메모

- 실제 파일 내용은 AdSense 퍼블리셔 ID 발급 후 확정한다.
- 기본 형식 예시:

```txt
google.com, pub-XXXXXXXXXXXXXXXX, DIRECT, f08c47fec0942fa0
```

- 퍼블리셔 ID가 없는 상태에서 임의 값으로 배포하지 않는다.

### 완료 조건

- 정책 페이지가 광고/데이터 처리/운영 신뢰 측면에서 초안 수준을 벗어난다.
- `ads.txt` 배포 위치와 운영 절차가 정리된다.
- 향후 CMP 적용이 필요한 지역을 고려한 문구와 구현 체크리스트가 준비된다.

### 예상 난이도

- 중간
- 정책 문안 정리 1일, 구현 0.5일, 검수 0.5일

## 4. 작업 스트림 3 — 카테고리 허브 페이지 추가

### 문제 정의

현재는 홈페이지와 각 툴 페이지만으로 사이트가 구성되어 있다. 검색 엔진과 사용자는 `developer`, `text`, `finance` 각각이 어떤 문제를 해결하는 영역인지 설명하는 랜딩 페이지를 보지 못한다. 이 구조는 내부 링크와 문맥 신호를 약하게 만든다.

### 목표

- 카테고리 자체를 설명하는 인덱스 페이지를 만든다.
- 카테고리와 개별 툴 간 관계를 강화한다.
- 향후 가이드/블로그/비교 페이지를 연결할 중간 허브를 확보한다.

### 작업 범위

1. `/developer`, `/text`, `/finance` 페이지 추가
2. 각 페이지에 아래 요소 포함
   - 카테고리 소개
   - 대표 사용 사례
   - 추천 툴 그리드
   - 카테고리 FAQ
   - 관련 가이드/향후 확장 영역
3. 헤더 네비게이션 연결
   - 드롭다운 라벨 클릭 시 허브 페이지 이동 가능하게 개선
   - 드롭다운 내 `View all` 또는 허브 링크 노출
4. 브레드크럼 개선
   - 현재 카테고리 라벨을 텍스트만 두지 말고 카테고리 허브 링크로 연결
5. 분류 재검토
   - `unit converter`, `data converter`, `date calculator`, `age calculator`의 카테고리 적합성 재평가
   - 즉시 변경이 어렵다면 허브 내 서브섹션으로 문맥 보완

### 신규/수정 대상 파일

- `src/app/developer/page.tsx`
- `src/app/text/page.tsx`
- `src/app/finance/page.tsx`
- 필요 시 `src/components/category/CategoryHubPage.tsx`
- `src/components/layout/Header.tsx`
- `src/components/layout/Breadcrumb.tsx`
- `src/components/tool/ToolPageLayout.tsx`
- 필요 시 `src/lib/tools-registry.ts`

### 허브 페이지 기본 레이아웃

```text
H1: [Category Name]
한 줄 설명

추천 사용 사례
대표 툴 카드 그리드

왜 FinePocket Toolbox의 이 카테고리를 써야 하는가
자주 찾는 작업 리스트
FAQ
향후 관련 가이드/비교 페이지 영역
```

### 구현 순서

1. 공통 허브 컴포넌트 설계
   - 카테고리 이름, 설명, 툴 목록, FAQ를 props로 받는 구조
2. `/developer`, `/text`, `/finance` 페이지 생성
3. 헤더 드롭다운 라벨을 허브 링크와 공존하도록 개편
4. 툴 페이지 브레드크럼의 카테고리 항목에 `href` 추가
5. 관련 툴 섹션과 허브 페이지 간 내부 링크 연결
6. 카테고리 설명 카피를 각기 다르게 작성
   - 단순 번역형 문구가 아니라 사용 의도 중심으로 차별화

### 완료 조건

- 각 카테고리에서 허브 페이지가 존재하고 직접 진입 가능하다.
- 개별 툴 페이지에서 카테고리 허브로 역링크된다.
- 헤더와 브레드크럼이 사이트 구조를 더 명확히 보여준다.

### 예상 난이도

- 중간
- 설계 0.5일, 구현 1일, 카피라이팅 및 검수 0.5일

## 7. 리스크와 의사결정 포인트

### 리스크

- 퍼블리셔 ID가 없으면 `ads.txt` 실제 배포는 완료할 수 없다.
- 금융 카테고리를 키울수록 정확성, 면책, 설명 책임이 커진다.
- 카테고리 재분류는 URL 구조까지 바꾸면 SEO 리스크가 생길 수 있으므로, 초기에는 허브 카피로 보완하는 편이 안전하다.

### 결정이 필요한 항목

1. 카테고리 재분류 범위
   - 지금은 허브 설명만 보강할지
   - 실제 `finance` 하위 URL 체계를 일부 조정할지
2. 운영자 정보 공개 수준
   - 개인 운영인지, 브랜드 운영인지
   - 공개 가능한 프로필/링크 범위
3. 광고 정책 강도
   - 비개인화 광고 우선인지
   - CMP 포함 개인화 광고까지 준비할지

## 9. 바로 구현할 때의 추천 순서

다음 순서로 실제 코드 작업을 진행하는 것이 가장 효율적이다.

1. 메타 중복 제거
2. `Privacy/About/Contact/Terms` 문안 확장
3. `ads.txt` 파일 경로 및 운영 메모 반영
4. 카테고리 허브 공통 컴포넌트 추가
5. `/developer`, `/text`, `/finance` 페이지 생성
6. 헤더/브레드크럼 연결
7. 최종 수동 점검

이 순서라면 심사 리스크를 먼저 줄이고, 그 다음 구조 개선으로 넘어갈 수 있다.
